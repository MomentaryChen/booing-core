# Code Review — cancelMyBooking 鎖定策略並發安全審查

**日期**：2026-04-15  
**範圍**：`ClientBookingService.cancelMyBooking` 改用 `findByIdAndPlatformUserIdForUpdate`；與 `createBooking` 死鎖風險評估；共享 service scope 殘留並發分析

---

## Findings

### [Critical] rescheduleMyBooking vs createBooking：確認死鎖路徑

**嚴重度**：Critical  
**來源**：此次提交引入 reschedule 路徑（與 cancel 共用鎖定入口），非 cancel 本身造成，但本次 review 需一併記錄。

**鎖定順序對比**：

| 交易 | Lock 1 | Lock 2 | Lock 3 |
|---|---|---|---|
| `createBooking` | `resource_items[R]` FOR UPDATE | `service_items[S1,S2,...]` FOR UPDATE | `bookings[overlap set]` FOR UPDATE |
| `rescheduleMyBooking` | `bookings[B_x]` FOR UPDATE | `service_items[S1,S2,...]` FOR UPDATE | `bookings[overlap set ∖ B_x]` FOR UPDATE |

**死鎖場景**：

```
Tx A (createBooking)      Tx B (rescheduleMyBooking)
-------------------------------------------------
LOCK resource_items[R]
                          LOCK bookings[B_x]
LOCK service_items[S]
                          → 等待 service_items[S]  ← A 持有
→ 等待 bookings[B_x]      ← B 持有
                          ↑ 循環等待 → DEADLOCK
```

`B_x` 只要在 createBooking 的 overlap scan 範圍內（相同 merchantId + serviceItemId + 時段重疊），此場景即可觸發。多租戶單一商戶高峰預約時尤為容易發生。

**建議修正**：reschedule 必須將 booking row 的鎖定移至 service items 鎖定之後，與 createBooking 保持相同鎖定順序：`service_items` → `bookings`。或改以 advisory lock / application-level serialized queue 依 `merchantId` 序列化此類操作。

---

### [Warning] cancelMyBooking vs createBooking：阻塞而非死鎖，但存在 WHERE 再評估依賴

**嚴重度**：Warning  
**評估結論**：**不會死鎖**，但需確認底層資料庫再評估行為。

**分析**：

- `cancelMyBooking` 僅獲取一個鎖：`bookings[B_x]` FOR UPDATE，之後不再請求任何其他鎖。
- `createBooking` 鎖定路徑：`resource_items[R]` → `service_items[S]` → `bookings[overlap 含 B_x]`。

因為 cancel 持有 B_x 後不需要任何 createBooking 所持有的資源，循環依賴不存在，**無死鎖**。

**需確認的行為**：createBooking 在 `findOverlappingBookingsForUpdate` 等待 B_x 鎖釋放後，PostgreSQL 會對已鎖定行重新評估 WHERE 謂詞（`b.status <> :excludedStatus`）。B_x cancel 後 status = CANCELLED，應被篩除，overlap 結果正確。

若使用 MySQL InnoDB，`SELECT ... FOR UPDATE` 在 REPEATABLE READ 隔離等級下**不保證**重新評估——等待後返回的行可能仍帶舊版本 status，導致誤判衝突。需確認 DB 及隔離等級。

### [Warning] resolveScopedServiceItemIdsForBooking：無鎖讀取 scope 再進行有鎖操作（TOCTOU）

**嚴重度**：Warning

```java
// rescheduleMyBooking 中呼叫
List<Long> scopedServiceItemIds = resolveScopedServiceItemIdsForBooking(booking);
// → 內部呼叫 resourceItemRepository.findByMerchantId(merchantId)：無鎖讀取
// → 之後才執行有鎖的 findByMerchantIdAndIdIn 與 findOverlappingBookingsForUpdate
```

若在讀取 scope 後、獲取 service items 鎖定前，有另一個交易修改了資源與服務的綁定關係，則 reschedule 使用的 scope 與鎖定對象不一致。可用性計算基礎失準。

**建議**：若需嚴格 scope 一致性，`findByMerchantId` 也應以 FOR UPDATE 讀取，或將 scope 解析移至獲取 booking 行鎖後且在同一 snapshot 內完成。

---

### [Warning] service_items 鎖定行序不受控制，可能引發同類交易間死鎖

**嚴重度**：Warning

`findByMerchantIdAndIdIn(merchantId, List<Long> ids)` 傳入的 `ids` 清單無排序保證，資料庫可能以不同順序獲取行鎖。兩個 createBooking（或兩個 reschedule）交易若服務範圍部分重疊（例如 `[S1, S2]` vs `[S2, S3]`），亦可能死鎖：

```
Tx A: LOCK S1, 等待 S2
Tx B: LOCK S2, 等待 S1
```

**建議**：在調用前對 `scopedServiceItemIds` 排序（`Collections.sort` 或 `.stream().sorted()`），確保所有交易以相同順序請求行鎖。

---

### [Warning] cancelMyBooking 缺少稽核日誌

**嚴重度**：Warning

`rescheduleMyBooking` 呼叫 `platformAuditService.recordForCurrentUser(...)` 記錄操作詳情，但 `cancelMyBooking` 沒有同等記錄。取消預約屬於高影響操作，應留下稽核軌跡（operationType、bookingId、reason、操作時間）。

---

### [Suggestion] listClientResources：全量載入無上限，scalability 隱患

**嚴重度**：Suggestion

```java
// listFeaturedResources
listClientResources(null, null, "rating", 0, Integer.MAX_VALUE).items();
// listClientCategories
listClientResources(null, null, "relevance", 0, Integer.MAX_VALUE).items()...
```

這兩個方法都呼叫 `listClientResources` 傳入 `Integer.MAX_VALUE`，會將全部商戶及其所有資源載入記憶體後再過濾。即使使用 `readOnly = true` 避免了寫鎖，當商戶數量增長時，這會導致 OOM 及高 GC 壓力。建議改用 DB 層分頁或聚合查詢。

---

### [Suggestion] appendCancellationReason 幂等保護過於簡單

**嚴重度**：Suggestion

```java
if (existing.contains("cancelReason=")) {
    return existing;
}
```

只要欄位曾寫過 `cancelReason=`，任何後續更新均被靜默丟棄。若業務允許更新取消原因，此邏輯會遮蔽變更且不報錯。需確認此為刻意的稽核保護（首次寫入不可覆蓋）還是邏輯缺陷，並加上明確注釋說明意圖。
