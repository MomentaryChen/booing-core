# Code Review Report

- **日期**: 2026-04-15
- **範圍**: Light review — `ClientBookingService`（`createBooking` / `listMyBookings` / `cancelMyBooking`）、`BookingRepository.findOverlappingBookingsForUpdate`
- **焦點**: CLIENT 範圍之安全／多租戶隔離、併發與狀態一致性

## 結論摘要

- **租戶／身分越權（CLIENT scope）**: 未發現可讀寫他人 `PlatformUser` 預約或繞過 `assertClientCanAccessResource` 之典型 IDOR 路徑。
- **仍須正視之問題**: 併發下之容量與狀態一致性（與「安全」邊界重疊處為 **商業／稽核風險** 與 **服務濫用**）。

---

## Findings（依嚴重度）

### Critical

1. **共享 service scope 之雙重預約窗口（併發正確性）**  
   - `createBooking` 先鎖 `ResourceItem`（`findWithLockById`），僅能序列化「同一 resource」之競爭。  
   - 註解宣稱鎖定「shared service scope」，但 `serviceItemRepository.findByMerchantIdAndIdIn` **未**宣告 `PESSIMISTIC_WRITE`。  
   - `findOverlappingBookingsForUpdate` 僅在「已存在重疊之 booking 列」時才會鎖到那些列；若目前尚無任何重疊列，查詢結果為空則 **可能未鎖定任何 booking 列**。  
   - 兩筆交易各鎖 **不同** `ResourceItem`、共用相同 `serviceItemIds` 時，可能皆通過重疊檢查後各自 `save`，導致同一時段／服務被重複預約（非 CLIENT 越權讀他人資料，但屬 **嚴重業務／租戶容量錯誤**）。

2. **取消與改期之交錯可能破壞狀態（無樂觀鎖＋取消未悲觀鎖）**  
   - `cancelMyBooking` 使用 `findByIdAndPlatformUserId`（無 `FOR UPDATE`）；`rescheduleMyBooking` 使用 `findByIdAndPlatformUserIdForUpdate`。  
   - `Booking` 實體 **無** `@Version`。  
   - 在特定交錯下，已提交之「已取消」可能被另一交易中仍載入舊狀態之改期流程以 UPDATE 覆寫（例如將 `status` 寫回非 `CANCELLED` 並套用新時段），造成 **幽靈預約／狀態機被繞過** 風險（需以 DB 隔離層級與 Hibernate flush 行為驗證；邏輯上為高風險）。

### Warning

3. **重疊判斷僅排除 `CANCELLED`**  
   - `excludedStatus` 為 `CANCELLED` 時，其餘列舉（如 `EXPIRED`、`REFUNDED`、`NO_SHOW` 等）若業務上應視為「不佔用容量」，可能 **誤擋** 或與營運預期不符（依產品定義調整）。

4. **`listMyBookings` 分頁條件使用 JVM `LocalDateTime.now()`**  
   - Specification 內之「現在」與 DB session 時區若不一致，極端情況下 tab 邊界可能與預期略有偏差（通常為低機率）。

### Suggestion

5. **註解與實作對齊**  
   - 若意圖以 service 列悲觀鎖序列化 scope，應在 repository 方法上補 `@Lock(PESSIMISTIC_WRITE)`（或等效之 advisory lock／可串行化隔離），否則應修正註解以免誤導維護者。

## CLIENT scope 安全／租戶觀察

| 項目 | 說明 |
|------|------|
| `createBooking` | `resourceId` 經鎖定後載入；`assertClientCanAccessResource` 隱藏 FORBIDDEN 為 404，降低列舉風險；`merchant` 來自 resource，非 client 可任意指定。 |
| `listMyBookings` | `ClientBookingSpecifications` 以 `platformUser.id` 內連結篩選，僅列本人預約。 |
| `cancelMyBooking` | `findByIdAndPlatformUserId` 綁定目前使用者，非本人為 404。 |

**無** 典型「跨使用者讀寫」或「任意 `merchantId` 注入」之 CLIENT API 路徑；主要殘餘風險在 **併發下容量與狀態**，而非身分偽造。

---

## Residual risks / Open questions

- **併發測試缺口**: 建議補整合測試（多 thread 或 DB 層）：(a) 兩 resource 共用 service、同一 `startAt` 並發 `createBooking`；(b) `cancelMyBooking` 與 `rescheduleMyBooking` 同一 `bookingId` 交錯。  
- **隔離層級**: 若未採 `SERIALIZABLE` 或明確鎖定策略，僅依賴現有 JPA 鎖可能不足。  
- **手動／後台建立之 booking**: 若與 client 流程共用同一重疊規則，後台路徑是否也遵守相同鎖定需一併檢視（範圍外略述）。
