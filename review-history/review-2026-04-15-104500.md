# Code Review Report

- Review Time: 2026-04-15 10:45:00
- Scope: `ClientBookingService` locking flow (`cancelMyBooking`, `rescheduleMyBooking`) and related repository lock queries
- Focus: concurrency safety, deadlock risk, performance, code quality

## Critical

- No critical findings.

## Warning

1. `PESSIMISTIC_WRITE` multi-row lock queries lack deterministic lock order, leaving residual deadlock probability under high contention.
   - `ServiceItemRepository.findByMerchantIdAndIdIn` (used by `createBooking` / `rescheduleMyBooking`) does not enforce `ORDER BY`.
   - `BookingRepository.findOverlappingBookingsForUpdate` also locks multiple rows without `ORDER BY`.
   - Even though `scopedServiceItemIds` are sorted before query, SQL `IN (...)` ordering does not guarantee row lock acquisition order in database execution plans.
   - Recommendation: add explicit ordered lock queries (e.g., `ORDER BY id ASC`) for all `FOR UPDATE` scans that may lock multiple rows.

2. `rescheduleMyBooking` computes `scopedServiceItemIds` by scanning all merchant resources each request, which can increase lock-hold time and contention.
   - Method `resolveScopedServiceItemIdsForBooking` calls `resourceItemRepository.findByMerchantId` and parses JSON for every reschedule.
   - Longer pre-lock computation inside transaction increases the chance of lock wait cascades in peak traffic.
   - Recommendation: precompute/normalize service-scope relations (mapping table or cached index) and keep lock section as short as possible.

## Suggestion

1. `cancelMyBooking` and `rescheduleMyBooking` both use booking row lock correctly; add integration tests that assert concurrent cancel/reschedule outcomes and lock timeout behavior.
2. Add operational metrics for lock waits / deadlocks (DB deadlock count, transaction wait duration) to validate behavior in production.

## Residual Risks

- If future code path introduces opposite lock order (booking first, then service rows), deadlock risk will rise quickly; lock order contract should be documented and enforced.
- Under very high write concurrency, pessimistic locking strategy may become throughput bottleneck; monitor before scaling traffic.
